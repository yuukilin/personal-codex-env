# TickTick Tracker -- 專案進度自動建入 TickTick

## Codex 相容性

- 本 skill 中的 `Claude` 指目前的 Codex 助手。
- Codex 目前不一定有 iOS `reminder_create_v0` 工具；若沒有，優先使用 Mac Reminders/AppleScript 路徑。
- 透過 AppleScript 寫入 Reminders 可能需要使用者授權；遇到權限限制時直接要求授權，不要假裝已建立。
- 若使用者其實要的是 Codex 自己日後提醒或定期追蹤，改用 Codex automations，不要混進 TickTick。

## 核心任務

使用者在對話中會討論各種個人專案、工作任務、或剛冒出的點子。
當使用者說「TickTick」時，Claude 執行以下流程：

1. **評估清晰度**：回顧本次對話，判斷使用者的目標和下一步是否足夠明確（見下方「清晰度判斷」）
2. **不夠明確 → 先問再建**：針對模糊的部分提問，釐清後再拆解並建立
3. **夠明確 → 直接拆解並建立**：運用任務拆解原則（見下方），將專案拆成可執行的子任務
4. **建完後告知**：簡短列出每個任務的標題、優先級、子任務數量

## 清晰度判斷

觸發 TickTick 後，Claude 先做一個內部檢查。以下任何一項不明確，就需要先問問題：

| 項目 | 明確的例子 | 不明確的例子 |
|------|-----------|-------------|
| 最終目標 | 「我要做一個 TAIFEX 選擇權波動率工具」 | 「我想弄個什麼東西追蹤市場」 |
| 範圍邊界 | 「只做 TXO，用 Python + Colab」 | 「可能做個工具吧」 |
| 成功標準 | 「能畫出波動率微笑曲線」 | 沒提到怎樣算完成 |
| 下一步行動 | 對話中討論了具體技術細節 | 只提了一個想法就沒了 |

### 提問原則

- 一次最多問 3 個問題，不要像審訊
- 問題要具體，給出選項讓使用者選，降低回答負擔
- 問完一輪就夠了，不要反覆追問。有些模糊是正常的，Claude 用合理假設補上，在 notes 中標註
- 問題聚焦在：你到底想達成什麼？範圍多大？有沒有時間壓力？

### 提問範例

使用者說：「我在想要不要來研究一下光通訊的設備商，ticktick」

Claude 判斷：目標模糊（研究什麼面向？要產出什麼？），需要釐清。

Claude 回覆：
「想確認幾件事再幫你拆：
1. 這個研究是要產出什麼？（例如：一篇 ELN 週報 / 一份完整框架分析 / 先建個觀察清單）
2. 範圍是鎖定哪些公司或哪個環節？（例如：InP 磊晶設備 / 矽光子封裝 / 還沒確定想先掃一輪）
3. 有時間壓力嗎？（例如：這週要交 / 慢慢來沒關係）」

## 任務拆解原則

這套原則來自生產力研究的核心發現，所有子任務都必須遵守：

### 原則一：每個子任務必須在 1 小時內能完成

如果一個子任務預估超過 1 小時，就要再拆。判斷方法：想像使用者坐下來開始做，能不能在 1 小時內做完並產出一個具體結果？

- 好的例子：「用 Bloomberg 拉出 Coherent / II-VI / Lumentum 近四季營收和毛利率，貼進 Google Sheet」
- 壞的例子：「研究光通訊產業鏈」（太大，不知道從哪開始）

### 原則二：子任務必須是「可直接動手」的動作

每個子任務要具體到使用者看到就知道要打開什麼工具、做什麼操作。
套用 implementation intention 的 if-then 邏輯：使用者看到這個子任務，不需要再思考「然後呢」，直接就能開始做。

- 好的例子：「打開 TradingView，把 COHR 和 LITE 的週線圖截圖，標註近半年支撐壓力位」
- 壞的例子：「分析股價走勢」（要分析什麼？用什麼工具？分析完要產出什麼？）

### 原則三：子任務之間要有邏輯順序

前一個任務的產出是下一個任務的輸入。使用者按順序做下來，自然就完成整個專案。
如果有些子任務可以平行做，在 notes 中標註「可平行」。

### 原則四：每個子任務要有明確的完成標準

使用者做完後要能明確判斷「這個做完了」。最好的完成標準是一個具體產出物。

- 好的例子：「完成後會有一份 Google Sheet，包含 5 家公司的營收、毛利率、EPS 數據」
- 壞的例子：「了解產業概況」（怎樣算了解？）

### 原則五：數量控制

- 一個專案的子任務：最少 3 個，最多 8 個
- 如果拆出來超過 8 個，把相關的子任務合併成階段性里程碑，每個里程碑建一筆獨立的 TickTick 任務
- 里程碑之間用命名標示順序，例如：「光通訊研究 (1/3) 數據收集」「光通訊研究 (2/3) 分析比較」

## 重要原則

1. **只整理本次對話的內容**，不搜尋過去對話。
2. **一次對話可能涉及多個專案**，每個專案建一筆獨立的提醒事項。
3. **語言用繁體中文**。
4. **觸發後的行為**：
   - 內容夠明確 → 直接拆解並建立，建完後告知
   - 內容不夠明確 → 先問問題，使用者回答後直接拆解並建立，建完後告知
5. 建完後的回報格式：「已建好 X 個任務，打開 TickTick 就會自動匯入。」並簡短列出每個任務的標題、優先級、子任務數量。

## 雙環境支援

Codex 需要偵測目前可用的工具，選擇對應的建立方式：

### 環境一：手機（iOS Claude app）
偵測方式：`reminder_create_v0` 工具可用。
建立方式：直接呼叫 `reminder_create_v0`，參數如下：
- `listId`：留空（使用預設清單，TickTick 只從預設清單匯入）
- `title`：專案名稱
- `priority`：high / medium / low
- `notes`：描述內容（格式見下方）

### 環境二：Mac 電腦
偵測方式：`reminder_create_v0` 不可用，但 `Control your Mac:osascript` 可用。
建立方式：使用 AppleScript 建立提醒事項，iCloud 同步後 TickTick 會自動匯入。
AppleScript 範本：

```applescript
tell application "Reminders"
    set targetList to default list
    set newReminder to make new reminder in targetList with properties {name:"任務標題", body:"描述內容", priority:優先級數字}
end tell
```

AppleScript 優先級數字對應：
- high（紅色）→ priority:1
- medium（黃色）→ priority:5
- low（藍色）→ priority:9

多個任務時，在同一個 tell block 內連續建立多筆 reminder。

### 環境三：都不可用
告知使用者：「目前環境無法自動建立提醒事項，以下是整理好的內容，請手動貼進 TickTick。」
然後輸出純文字格式供使用者複製。

## notes 描述格式

所有任務統一使用以下格式：

```
目標：[一句話說明這個專案要達成什麼]
進度：[目前做到哪裡，用 1-3 句話。如果是全新想法，寫「尚未開始」]
卡點：[目前卡在什麼地方，如果沒有就寫「無」]
更新：[今天的日期，格式 YYYY-MM-DD]

- [ ] 子任務 1（動詞開頭 + 具體工具/對象 + 預期產出）
- [ ] 子任務 2（動詞開頭 + 具體工具/對象 + 預期產出）
- [ ] 子任務 3（動詞開頭 + 具體工具/對象 + 預期產出）
...

#Claude
```

### 格式規則

1. **標籤固定為 `#Claude`**，放在 notes 的最底部。所有經由此 Skill 建立的任務都用這個標籤。
2. **更新日期**：每次建立都加上當天日期。
3. **子任務寫法**：
   - 用 `- [ ]` 格式，TickTick 會自動辨識為可打勾的 checklist
   - 每個子任務遵守上方「任務拆解原則」的五條規則
   - 格式：「動詞開頭 + 具體工具/對象 + 預期產出」
   - 好的範例：
     - 「用 Bloomberg 拉出 COHR / LITE / IIVI 近四季財報數據，整理進 Google Sheet」
     - 「在 Seeking Alpha 搜尋近一個月 InP 相關 earnings call 提及，截圖 3-5 段關鍵內容」
     - 「打開 Obsidian，建立 [[光通訊設備商]] 筆記，寫入初步產業地圖和公司清單」
   - 壞的範例：
     - 「研究產業」
     - 「繼續分析」
     - 「收集資料」

### 子任務拆解的內部思考流程

Claude 在寫子任務前，先在內心跑一遍這個檢查：

1. 使用者的最終目標是什麼？
2. 要達成這個目標，有哪些主要階段？（top-down 拆解）
3. 每個階段裡，具體要做哪些動作？（bottom-up 細化）
4. 這些動作之間的依賴關係是什麼？哪些要按順序，哪些可以平行？
5. 每個動作都在 1 小時內能完成嗎？完成後有具體產出嗎？
6. 使用者看到這個子任務，不用再想就能直接動手嗎？

## 優先級判斷邏輯

- **high**（紅色）：這次對話中正在積極推進、使用者明確表示要做的
- **medium**（黃色）：之前有做過但這次只是順帶提到、或使用者說「之後再弄」
- **low**（藍色）：剛冒出來的新點子、還在評估要不要做的

## 邊界情況處理

- 如果這次對話只是在問問題、沒有明確的專案推進，告知使用者「這次對話沒有明確的專案進度需要記錄」。
- 如果使用者在對話中討論了多個主題但只有部分是「專案」性質的，只整理專案相關的部分。
  判斷標準：有沒有「要做出什麼東西」或「要達成什麼目標」。單純的知識問答不算專案。
- 如果專案名稱不明確（使用者沒有給名字），Claude 自己取一個簡短有辨識度的名稱。
- 如果同一個專案可能在之前的對話中已經建過，在建完後的回報中提醒使用者：「注意：這個專案可能之前已建過，如有重複請在 TickTick 中合併或刪除舊的。」
- 如果一個專案拆出超過 8 個子任務，拆成多筆 TickTick 任務（見上方「數量控制」）。
